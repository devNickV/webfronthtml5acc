JSHint

One of the godfathers in the javascript community and javascript architect at Yahoo, Douglas Crowford, ones wrote jslint. That was a configurable parser/code checker. It contained a very opinionated view on which javascript contructs should and should not be used. To see it work http://www.jslint.com/.

JavaScript parser that can analyze a JavaScript program and report on the bad parts that it contains. JSLint pro- vides a degree of rigor that is generally lacking in JavaScript development. It can give you confidence that your programs contain only the good parts.

Douglas Crowford is also writer of the rather small book: "javascript the good parts". A easier configurable and runner http://www.jshint.com/. The disadvantage of this tool is it is not configurable as a commandline tool.Therefore a new tool was build called jslint. linkie_todo. This jslint solution is fully integratable in your workflow.

Debuging

Once there was firebug...oh new addon/plugin for firefox. This changed the world. With the tool you were able to:
  • inspect the live situation in a page
  • introspect the DOM
  • change DOM attributes on the fly
  • see the performance and network traffic of your page
  • etc. etc

Currently all main browser have these capabilities. The most used seems to be chrome at this moment.

Excercise: take the development view of chrome, IE or firefox and get familiar with these capabilities. Load some of your favourite websites and see how they perform, how the DOM looks like and ... For a far more detailed explanation of the possibilities within chrome see. As being said on the page. Familiarity with this tool is almost neccessary from a front-end developer.

Workflow / automating task

This were quite a few tools and technics. We need to make this tools work together in an automated fashion as much as possible. The predomenantly justed tool for running tasks is grunt. In the minimal workflow we want to start with
